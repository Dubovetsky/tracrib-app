# 15. ASR Runtime Contract

These rules are production constraints for the transcription pipeline. They are not UI labels.

- Default production transcription is verbatim-first: `PRESERVE_ASR_WORDS=1`. Hidden rewriting of ASR words is a quality defect.
- Completed jobs must expose operability metadata through the API: `diarization_status`, `speaker_count`, `warnings`, phase `timings`, and pipeline `diagnostics`.
- Completed jobs must preserve immutable raw ASR text before speaker assignment or postprocessing: `raw_asr.txt`. RAW must be downloadable as soon as it exists, including while a job is still processing.
- Each job must have its own `job.log` under `backend/data/results/{job_id}/job.log`; shared backend logs are not enough for user-facing diagnostics.
- `/api/jobs/{job_id}/download/logs` must be available for successful, failed, cancelled, and still-running jobs when a job log exists.
- Production jobs should run in isolated worker subprocesses by default (`JOB_SUBPROCESS_ENABLED=1`) so cancel can terminate the worker process tree.
- Audio preprocessing must use `ffmpeg -nostdin` and bounded timeouts so backend workers do not hang waiting for terminal input.
- ETA must be derived from selected pipeline, actual provider path, hardware/profile, and calibration from matching completed jobs. UI must not present a static time budget as ETA.

## Recognition Mode Contract

Modes are separate pipelines, not module toggles.

- `fast`: `diarization_mode=none`; ASR-only draft path; no pyannote, no heavy speaker pipeline, no cloud/local text polish; RAW should appear first.
- `balanced`: `actual_pipeline=balanced_text_analysis`; `diarization_mode=lightweight`; conservative audio preprocessing; stronger ASR decode than `fast`; glossary prompt/hotwords and word timestamps enabled; text cleanup/polish and bounded lightweight speaker-structure heuristics allowed; no full pyannote and no hidden maximum fallback.
- If `balanced` has a configured text-polish provider, it behaves like the old medium path: ASR plus analysis/structuring/polish for long recordings.
- If `balanced` has no configured provider and falls back to local rules, diagnostics must explicitly show `text_analysis_status=local_fallback`, and ETA must estimate that faster fallback path honestly.
- `accurate`: `diarization_mode=full`; full acoustic diarization is allowed and expected to be slower.
- Historical jobs with full diarization timings must not calibrate draft-mode ETA.
- A bounded pipeline that cannot complete optional speaker/text-analysis work must finish with an explicit fallback instead of drifting into a slower mode.

## Speaker And Text Quality Rules

- Diarization failures may fall back, but they must be visible as warnings and diagnostics. A successful-looking transcript with hidden diarization failure is a production defect.
- Text-only speaker extraction must reject obvious garbage labels such as `По:`, `Кто:`, `Какая:`, `Как:`, `Из:`, and `Pmi:`. These strings must stay in transcript text, not become speaker identities.
- Diarization should prefer word-level splitting when ASR words are available; segment-level overlap is only a fallback.
- Upload flow should accept per-job expected speaker count, participant names, and custom vocabulary. These must affect ASR/diarization inputs instead of relying on one generic deployment glossary.
- Cloud text polish may fix spelling, punctuation, casing, Russian/English words, and abbreviations, but must not summarize, omit facts, add facts, or change meaning. Segment order, timestamps, and speaker labels must be preserved.

## UI Processing Contract

- Processing UI must block file selection and recognition-quality selection while a job is queued or processing.
- The active history item must not show a destructive delete control.
- Deleting a history item must require an explicit confirmation overlay.
- Runtime diagnostics should show the backend's actual behavior: selected mode, actual pipeline, fallback status, speaker separation status, text analysis status/provider, elapsed time, warnings, and downloadable logs.
